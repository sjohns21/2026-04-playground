02 · JWT Anatomy

A JSON Web Token is three Base64URL-encoded strings joined by dots. Each part has a distinct role — click a panel to explore it.

Header
Declares the token type and signing algorithm.
"alg": "HS256"
"typ": "JWT"
Payload
Carries claims — facts about the user and token lifecycle.
"sub": "user_123"
"name": "Alice"
"role": "admin"
"iat": 1700000000
"exp": 1700003600
Signature
Cryptographic proof that the token hasn't been tampered with.
HMACSHA256( base64(header) + "." + base64(payload) , secret )
Why it works

The header and payload are just Base64URL-encoded — anyone can read them. The signature is what makes the token trustworthy: it's computed using a secret key only the auth server knows. If an attacker modifies any claim in the payload, the signature won't match and the resource server rejects the request.

03 · Key Concepts
🎫
Bearer Token
Whoever holds the token can use it. Sent in every API request via the Authorization: Bearer <token> header.
📋
Claims
Key-value facts embedded in the payload. Standard claims: sub (subject), iat (issued at), exp (expiration). You can add custom ones.
⏳
Expiration
JWTs have a short lifetime (minutes to hours). After exp, the token is invalid — limiting damage if it's stolen.
🔑
Signing Keys
HS256 uses a shared secret. RS256 uses a private key to sign, public key to verify — better for distributed systems.
♻️
Refresh Tokens
A long-lived token stored securely. Used to fetch a new access token when the old one expires — without asking the user to re-login.
🔒
HTTPS Only
Tokens are credentials. Always transmit over TLS. A token captured over plain HTTP gives an attacker full access until expiration.
